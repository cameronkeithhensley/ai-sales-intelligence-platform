# Database

> Redacted reference for the data layer. Column descriptions are included for **public (cross-tenant) tables only**. Per-tenant schema DDL is intentionally excluded.

---

## Engine and extensions

- **PostgreSQL** on AWS RDS.
- **PostGIS** extension for geographic queries on tenant-scoped data.
- `pgcrypto` for `gen_random_uuid()`.
- Single logical database per environment (`intent_signaler` in the live system; any name is fine here).

---

## Multi-tenancy — schema-per-tenant

Tenant data is isolated by PostgreSQL schema, not by a `tenant_id` filter on shared tables.

- **Shared schema**: `public` — contains cross-tenant tables (below).
- **Per-tenant schemas**: one schema per tenant, created at onboarding, dropped at offboarding. All per-tenant tables have identical DDL across tenants.
- **Role model**: each tenant has a PostgreSQL role whose `search_path` is scoped to its own schema plus the minimum `public` tables it needs to read. Tenant roles cannot see other schemas at all.
- **Mapping**: the `tenants` table maps Cognito `sub` → tenant UUID → schema name. Every service resolves the caller's tenant from the JWT before touching data.

### Why schema-per-tenant

- A forgotten `WHERE tenant_id = …` clause cannot leak data — the role physically cannot reach another schema.
- Offboarding is a single `DROP SCHEMA CASCADE` with zero residual risk.
- Per-tenant schema changes (rare) can be rolled individually.

### Trade-offs

- Schema count grows linearly with tenants. Comfortable at low thousands of tenants on a single RDS instance; beyond that the pattern needs revisiting (sharding, or a move to row-level security).
- Cross-tenant analytics require privileged roles that read across schemas explicitly.

## Public tables

These tables are shared across all tenants and accessed by every service. Per-tenant tables are intentionally not documented here.

### `public.tenants`

Master tenant registry. Maps an authenticated identity to a tenant row and its dedicated schema.

| Column | Description |
|---|---|
| `id` | UUID primary key. Referenced by every other table's `tenant_id`. |
| `name` | Human-readable tenant name for operator surfaces. |
| `tier` | Subscription tier (enum). Drives quota enforcement at the service layer. |
| `tenant_type` | High-level tenant category that gates which agents and data sources apply. |
| `cognito_sub` | OIDC subject from Cognito. Used at request time to resolve the caller's tenant. |
| `schema_name` | Name of the per-tenant PostgreSQL schema. |
| `created_at`, `updated_at` | Lifecycle timestamps. |

### `public.jobs` (a.k.a. `job_results`)

Async agent job tracking. Every agent writes status and output here; callers poll by `job_id`.

| Column | Description |
|---|---|
| `job_id` | UUID primary key. Generated by the producer at enqueue time. |
| `tenant_id` | FK to `tenants.id`. Every job belongs to exactly one tenant. |
| `agent` | Which agent role owns this job (enum). |
| `subject_type` | What the job is about at a high level (e.g. a company, a property, a person). |
| `status` | `pending` → `running` → `completed` \| `failed`. |
| `result` | JSONB output payload. Schema varies by agent and subject type. |
| `error_message` | Populated on failure. |
| `policy_version` | The version of the policy rules in effect when the job ran. Used for audit replay. |
| `enqueued_at`, `started_at`, `completed_at` | Lifecycle timestamps. |
| `created_at` | Row creation time. |

Indexes cover the two dominant read patterns: by tenant, and by `(tenant_id, status)` for "show me all pending jobs for this tenant."

### `public.audit_log`

Append-only cross-tenant policy audit. Every policy-enforcement decision is recorded — allowed, blocked, or flagged.

| Column | Description |
|---|---|
| `id` | UUID primary key. |
| `tenant_id` | FK to `tenants.id` (nullable for system-level events). |
| `user_id` | The acting user within the tenant, when applicable. |
| `job_id` | The job this decision relates to, when applicable. |
| `agent` | Which agent evaluated the decision. |
| `action` | Short machine-readable action name. |
| `subject_type` | What kind of subject the decision applied to. |
| `source_legality_tier` | Legality classification of the data source, if any. |
| `policy_version` | Policy version in effect. |
| `outcome` | `allowed` \| `blocked` \| `flagged`. |
| `metadata` | JSONB payload with decision context. |
| `created_at` | Event time. |

**Tenant-scoped database roles do not have `DELETE` on this table.** Policy events are preserved even under SQL misuse.

### `public.outreach_events`

Log of every outbound customer-facing message attempt — the CAN-SPAM / telecom-compliance audit trail.

| Column | Description |
|---|---|
| `id` | UUID primary key. |
| `tenant_id` | FK to `tenants.id`. |
| `contact_type` | `email` \| `sms` \| `whatsapp`. |
| `contact_value` | The destination address or number, as sent. |
| `event_type` | `sent`, `blocked`, `bounce_hard`, `bounce_soft`, `complaint`, `delivered`, `open`, `click`. |
| `provider_message_id` | Upstream message id returned by the delivery provider, when available. |
| `subject_line` | For email events. |
| `metadata` | JSONB payload with provider-specific fields. |
| `created_at` | Event time. |

Indexed for per-tenant time-series reads, per-contact history lookups, and provider message id reverse-lookups for webhook processing.

## What is not in this document

- Per-tenant schema DDL (prospect records, conversation history, per-tenant configuration).
- Signal-type catalogs or strength weights.
- Migration sequencing and ordering details beyond what is visible from file names in the `migrations/` directory.
